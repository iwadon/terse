# Terse 再設計プロポーザル

> **Status: Draft（主要な設計判断は確定済み）**

## 0. 背景と動機

現在の terse は POSIX テキスト UI のための統合ライブラリとして機能しているが、以下の構造的な課題が見えてきている:

- `src/terse.c` が「ケイパビリティ検出」「エスケープ生成」「状態管理」「イベントパース」などを抱えており、責務が広い
- P0-P3 プロファイルは概念として分かりやすいが、粒度が粗く「P2 を要求したらマウスもタイトルも全部欲しい」という前提を強いる
- プロファイルの色分類が「16 / 256 / TrueColor」しかなく、**それ以下の段階を表現できない**。具体例として Human68k (X68000) はテキスト表示で 4 色が利用可能で SGR エスケープ (`^[[Nm`) も受け付けるが、4 色という分類が存在しないため色指定 API を有効化できず、現状では色なし扱いとなっている
- 描画モデルが「即時エスケープ出力型」のため、ちらつき抑制や部分更新の最適化を後付けしづらい
- テスト戦略は API call recording 中心で、「呼び出し順序」は検証できても「出力バイト列」や「実端末上の見え方」までは保証できない
- 関連プロジェクトである [`terse-prompt`](https://github.com/iwadon/terse-prompt) でセルベース仮想バッファ + 差分描画が既に実装されており、設計知見の取り込み余地がある

本プロポーザルでは、これらに対する再設計の方向性を提示する。一気にゼロから書き直すのではなく、**段階的に近づけていく**ことを前提とする。

## 1. 全体方針

### 1.1 三層分離 + ケイパビリティ中心

```
┌───────────────────────────────────────────────┐
│  terse-ui    (将来構想: ウィジェット、レイアウト)        │  ← 別ライブラリ
├───────────────────────────────────────────────┤
│  terse       (中レベル: イベントループ、描画バッファ)     │
├───────────────────────────────────────────────┤
│  terse-term  (低レベル: 端末I/O、エスケープ生成、検出)    │
├───────────────────────────────────────────────┤
│  terse-sys   (プラットフォーム抽象: termios/Win32/...)   │
└───────────────────────────────────────────────┘
```

- **terse-sys**: 既存の `src/terse_platform.h` 相当。OS 固有の I/O・termios・コンソールモード
- **terse-term**: ケイパビリティ検出、エスケープシーケンス生成、入力イベントパース
- **terse**: セルベース仮想バッファ、状態管理、イベントループ統合
- **terse-ui**: （将来）ウィジェット、レイアウト、フォーカス管理。今回のスコープ外だが将来の収まりとして意識しておく

### 1.2 主な変更点

| 項目 | 現状 | 再設計後 |
|------|------|----------|
| 層構造 | `terse.c` に集約 | `terse-term` / `terse` / (将来 `terse-ui` は別ライブラリ) に分離 |
| 機能の選択 | P0-P3 プロファイル中心 | ケイパビリティ宣言中心、プロファイルはプリセット |
| 描画モデル | 即時エスケープ出力 | セルベース仮想バッファ + 差分出力（オプトイン） |
| イベント待機 | `terse_read_event()` のみ | コア API + POSIX 拡張ヘッダで `pollfd` を露出 |
| テスト | API call recording | バイト列ゴールデン + PTY 実プロセステスト |
| 内部エンコーディング | UTF-8 固定（現状維持） | UTF-8 固定（現状維持） |

## 2. 各論

### 2.1 ケイパビリティ中心への移行

> **note**: 調査の結果、terse は既に細かいケイパビリティ構造体 (`terse_capabilities_t`、20以上のフィールド) を内部で持っており、プロファイルはその上に被せた「便利な束ね」として機能している。したがって本セクションの本質は **「ゼロからケイパビリティ API を作る」ではなく「既存ケイパビリティに要求 API を被せ、散在を整理し、段階を追加する」** ことである。

#### 問題
現状の P0-P3 は「マウスが欲しいなら P2 全部」を強要する。実際には「TrueColor は欲しいがマウスは要らない」「マウスは欲しいがタイトルバー操作は要らない」といった要求が出る。

加えて、既存の `terse_capabilities_t` には以下の整理しきれていない箇所がある:
- 色関連のフィールドが `has_sgr_basic` / `has_sgr_extended` / `has_truecolor` / `colors` の 4 箇所に散在
- 同じ概念が複数フィールドにまたがり、関係性が不明瞭

さらに、**色の分類粒度が粗い**ことも問題になる。現状の色ケイパビリティは「16 / 256 / TrueColor」で、それ以下の段階を表現できない。例えば:

- **Human68k (X68000)**: テキスト表示で 4 色が利用可能で、SGR エスケープ (`^[[Nm`) も受け付ける。しかし「4 色」という分類が無いため色指定 API を有効化できず、結果として利用側で色指定を外す形になっている
- 将来サポートする可能性のある旧型端末や組み込み環境でも、似た「VT は通るが色数が限られる」状況が起こりうる

色数の段階を細かく刻めるようにケイパビリティを設計しておけば、こうしたプラットフォームを取りこぼさずに済む。

#### 提案
1. **要求 API を追加**: アプリは必要なケイパビリティを宣言する
   ```c
   terse_require(handle, TERSE_FEAT_MOUSE | TERSE_FEAT_TRUECOLOR);
   uint64_t missing = terse_caps_missing(handle);  // 端末が提供できなかったもの
   ```
2. **プロファイルはプリセット化**: P0-P3 は「便利なケイパビリティ束」として公開 API に残す
   ```c
   terse_require_profile(handle, TERSE_PROFILE_P2);  // 内部的に該当ケイパビリティ群を展開
   ```
3. **検出結果と要求を分離**: `terse_get_capabilities()` は「端末が提供可能なもの」、`terse_get_active_features()` は「現在有効化されているもの」を返す

4. **色数の段階を細かく刻む**: ケイパビリティとしての色を「4 色 / 16 色 / 256 色 / TrueColor」のように段階化し、プラットフォームごとに該当する上限を登録できるようにする。詳細は [4.2.2](#422-色数の段階分け確定) 参照。

#### Human68k の具体的な扱い（例示）
ケイパビリティ中心設計のもとで、Human68k は以下のように表現される:

- `colors`: `TERSE_COLOR_BASIC4`（SGR エスケープ経由、テキスト表示の 4 色）
- `has_sgr_basic`: 有効
- `has_sgr_extended`: 無効
- `has_truecolor`: 無効
- `mouse`: 無効（または IOCS マウス経由で実装可能なら有効）
- `has_move_absolute` / `has_move_relative`: 有効
- `has_title`: 無効

これにより、アプリは「4 色を要求」とだけ宣言すれば、Human68k でも色出力が有効化される。出力経路自体は既存の SGR 出力をそのまま流用できる。

#### トレードオフ
- (+) 既存のプロファイル API は残せるので後方互換性を保ちやすい
- (+) 利用側がやりたいことを直接表現できる
- (+) **色数粒度が細かいプラットフォームを正当に評価できる**（Human68k の 4 色がついに使えるようになる）
- (−) ケイパビリティフラグの数が増えるので、定数の体系化が必要
- (−) 「このケイパビリティ群はセットで意味を持つ」みたいな暗黙の依存関係をどう扱うか要検討（例: マウスSGRエンコーディングはマウス本体が前提）

#### 関連する確定事項
- ケイパビリティの粒度: [4.2.1](#421-ケイパビリティフラグの粒度確定) — 現状の粒度を保ちつつ整理
- 色数段階: [4.2.2](#422-色数の段階分け確定) — NONE / 4 / 16 / 256 / TrueColor の 5 段階
- プロファイル API の去就: [4.1](#41-スコープ判断) — Phase 4 完了後に判断（保留）

### 2.2 描画モデル: セルベース仮想バッファの導入

#### 問題
即時出力モデルは単純だが、以下が苦手:
- ちらつき抑制（同じ位置に何度も書くと毎回エスケープが飛ぶ）
- 部分更新の最適化
- テスト時の「最終的な画面状態」のアサート

#### 提案
**オプトイン**でバッファドモードを追加。即時モードは現状維持。

```c
terse_open(&opts);                     // 既定: 即時モード
opts.render_mode = TERSE_RENDER_BUFFERED;
terse_open(&opts);                     // バッファドモード
```

バッファドモード時:
- `terse_write_text` 等は内部の仮想セルバッファに書き込むだけ
- `terse_flush()` でバッファ間 diff を計算 → 変更セルだけエスケープ出力
- ダブルバッファ（current / previous）+ ダーティセル配列

#### terse-prompt からの取り込み

`terse-prompt` には既にセルベース実装が存在する（`src/tprompt_display.c` 約1500行、`src/tprompt_internal.h` の `tprompt_screen_cell_t` / `tprompt_screen_buffer_t`）。これを **設計借用で terse 用に再実装**する（詳細は [4.3.1](#431-描画コードの取り込み方)）。

借用できる要素（ロジック）:
- ダブルバッファ + diff アルゴリズム
- リサイズ対応
- セル書き込み・差分検出のロジック

terse 向けに再設計する要素:
- セル構造体: `terse_color_t` の fg/bg + effects を含む形（詳細は [4.2.3](#423-バッファドモードでの色属性表現セル構造体のレイアウト確定)）
- 端末ハンドル依存を `terse_handle_t` に変更
- SGR 出力（色対応）の追加

#### terse-prompt との関係
取り込み後、`terse-prompt` 側は描画ロジックを捨てて terse のバッファ層を **使う側** にリファクタリングする（[4.3.2](#432-terse-prompt-側のリファクタリングのタイミング) 参照。Phase 5 で API 検証、Phase 5.5 で本格移行）。

#### トレードオフ
- (+) ちらつき削減、テスト容易性、将来の TUI 機能の土台
- (+) `terse-prompt` のコード量削減（`tprompt_display.c` 汎用部分の 500-700 行を削除可能）
- (−) 即時モードと両立するため API が二系統になる（オプション切り替えで吸収する想定）
- (−) メモリ使用量増（端末サイズ × セル構造体 × 2）

#### 関連する確定事項
- セル構造体: [4.2.3](#423-バッファドモードでの色属性表現セル構造体のレイアウト確定)
- alt screen 対応: [4.2.4](#424-alt-screen--scrollback-の扱い確定) — terse 中レベル層で提供（低レベル API + モード化両方）
- スクロール領域抽象化: 本プロポーザルのスコープ外（必要が出てから検討）

### 2.3 イベントループ統合（POSIX 拡張）

#### 問題
現状は `terse_read_event()` をユーザーがループで呼ぶ。複数 fd（端末入力 + ソケット + タイマー）を一つの poll/epoll/kqueue ループで待ちたい場合に統合できない。

#### 提案
1. **コア API は OS 非依存のまま**: `terse_wait_event(handle, timeout_ms)` を維持
2. **POSIX 拡張ヘッダを追加**: `<terse/posix.h>` 経由で `pollfd` を露出
   ```c
   #include <terse/posix.h>
   int fd = terse_posix_get_input_fd(handle);
   // ユーザーの poll() ループに組み込む
   ```
3. Windows / Human68k では別の同等手段が必要なら、各々の拡張ヘッダで（`<terse/win32.h>` 等）

#### トレードオフ
- (+) コア API は OS 非依存を保てる
- (+) 必要な人だけ拡張ヘッダを include する
- (−) プラットフォームごとに拡張ヘッダを設計する必要がある
- (−) Windows 側で「fd 相当」をどう公開するか要検討（`HANDLE` か、または別の抽象）

#### 関連する確定事項
- Windows での POSIX 相当機能: [4.2.5](#425-windows-での-posix-相当機能の提供範囲確定) — 初期版は POSIX のみ、Windows は後追い

### 2.4 テスト戦略の刷新

#### 問題
現状の `TERSE_ENABLE_TEST_MODE` による API call recording は「terse が正しい API を呼んだか」までしか検証できず、「最終的に端末に何が出力されたか」「実端末でどう見えるか」までは保証されない。

#### 提案
三段構えのテスト階層:

1. **ユニットテスト**: 個別関数（既存と同等）
2. **バイト列ゴールデンテスト**: API 呼び出し → 出力バイト列をキャプチャ → ゴールデンファイルと比較
   - エスケープシーケンスの正しさを検証
   - ケイパビリティ別シナリオ（TrueColor 有効 / 256色フォールバック 等）
3. **PTY 実プロセステスト**: テストバイナリを PTY 配下で起動 → 入出力を検証
   - CI 標準で動かす
   - リサイズ・シグナルなど実環境でしか出ない挙動をカバー

バッファドモード導入後は **(4) 仮想セルバッファのスナップショットテスト** も追加できる（バッファ状態を直接アサート）。

#### トレードオフ
- (+) 実出力レベルでの保証が得られる
- (+) リグレッション検出力が大幅に上がる
- (−) ゴールデンファイルの保守コスト（端末ごとの差異をどう扱うか）
- (−) PTY テストは CI 環境依存（GitHub Actions ランナー / Windows での扱い）

#### 関連する確定事項
- ゴールデンファイル運用: [4.4.1](#441-ゴールデンファイルの保守フロー) — ハイブリッド方式
- PTY テストの Windows 対応: [4.4.2](#442-pty-テストの-windows-対応優先度) — POSIX のみ、Windows は後追い
- スナップショットの粒度: [4.4.3](#443-端末別スナップショットのカバレッジ) — ケイパビリティ組み合わせベース

### 2.5 内部エンコーディング（現状維持）

UTF-8 固定は既に達成されている。`mini_iconv` は外部境界での Shift_JIS ↔ UTF-8 変換専用として現状維持。

検討対象として残るもの:
- East Asian Width テーブルの Unicode バージョン明示と自動生成化
- 絵文字の ZWJ シーケンス対応（必要になったら）

これらは今回のスコープではなく、別途扱う。

## 3. 移行ロードマップ

「いきなり全部やる」ことはしない。以下の Phase 順で進める。各 Phase は独立してリリース可能なサイズに収める。

### Phase 1: ディレクトリ分離（破壊的変更なし）
- `src/term/` と `src/core/` にファイルを物理的に分割
- 公開ヘッダ `include/terse.h` は変更しない
- 関数の所属が可視化されるのが主な成果
- **影響範囲**: ビルドシステムのみ。利用側コードへの影響なし

### Phase 2: 内部ヘッダの分離（破壊的変更なし）
- `src/terse_term_internal.h` を新設し、低レベル層が公開する関数を明示
- 中レベル層は内部ヘッダ経由でしか低レベル層を呼ばないように整理
- **影響範囲**: 内部のみ

### Phase 3: 公開ヘッダの分離（小規模な破壊的変更）
- `include/terse/term.h` を新設、低レベル層 API を切り出す
- `include/terse.h` は `terse/term.h` を include する形にして既存利用側との互換を保つ
- ユーザーは初めて「低レベル層だけ使う」選択肢を持てる
- **影響範囲**: include path の見直しが必要な利用側があるかもしれない

### Phase 4: ケイパビリティ API の拡充（追加のみ、破壊的変更なし）
- `terse_require()` / `terse_caps_missing()` / `terse_get_active_features()` を追加
- 既存のプロファイル API は内部でケイパビリティ展開する形に置き換え
- 色 API を「経路非依存」の形に整理（`terse_set_fg_color` 等）
- **影響範囲**: 利用側コードは無変更で動く。新 API は opt-in

### Phase 4.5: Human68k テキスト 4 色対応（検出ロジックの修正が主）
- Phase 4 のケイパビリティ基盤を前提に、Human68k を「色 4 / SGR 経由」として登録
- 色出力は既存の SGR 経路 (`^[[Nm`) をそのまま使用
- 利用側で色指定を外している箇所を解除し、ケイパビリティに従って自然に有効化されることを確認
- **影響範囲**: Human68k 環境および同等の「色数が限られた SGR 対応端末」
- **メリット**: ケイパビリティ中心設計の価値を実証する最初のユースケース。新基盤の検証も兼ねる

### Phase 5: バッファドモードの導入（追加のみ、破壊的変更なし）
- `terse-prompt` の描画コードを参考に、セルベース仮想バッファを実装
- `terse_open()` の opts に `render_mode` を追加
- `terse_options_t` に `use_alt_screen` フラグ、低レベル API `terse_enter_alt_screen()` / `terse_leave_alt_screen()` を追加（[4.2.4](#424-alt-screen--scrollback-の扱い確定)）
- `terse-prompt` 側で最小限のプロトタイプ適用を行い、API を検証（本格移行は Phase 5.5）
- **影響範囲**: 既存利用側は即時モードのまま動く

### Phase 5.5: terse-prompt の本格リファクタリング（terse-prompt 側の作業）
- terse-prompt の `tprompt_display.c` 汎用部分（500-700 行）を削除
- terse の新バッファ層を **使う側** に置き換え
- terse-prompt の独自セル構造体を撤去
- テストは観点を参考に terse 側で新規作成済み（[4.3.3](#433-テストの扱い)）。terse-prompt 側はプロンプト固有機能テストのみ残す
- **影響範囲**: terse-prompt のみ。terse 本体への影響なし

### Phase 6: テスト戦略の刷新
- バイト列ゴールデンテストの基盤導入
- PTY テストの導入
- 既存テストは段階的に移行

### Phase 7: POSIX 拡張ヘッダの追加
- `<terse/posix.h>` で `pollfd` を露出
- Windows 拡張は後追い検討

### Phase 8 以降: ハンドル分離（**現状はスコープ外**）
- `terse_handle_t` を `terse_term_t` と `terse_t` に分割し、`terse_t` が `terse_term_t` を内包する構成
- 公開 API の大規模な破壊的変更を伴う（既存利用側コードの全面書き換えが必要）
- **本プロポーザルのスコープには含めない**。理由:
  - Phase 1-7 で達成したい主な目的（層の可視化、ケイパビリティ中心、バッファド描画、テスト刷新）はハンドル分離なしでも実現可能
  - 破壊的変更のコストに見合う具体的なユースケース（低レベル層単独利用の強い需要等）が現時点では明確でない
- 将来「ケイパビリティ検出だけ使いたい」「エスケープ生成だけ使いたい」といった具体的な需要が出てきた場合に、改めて別プロポーザルとして検討する

## 4. オープンクエスチョン

合意が必要な項目を以下に列挙する。

### 4.1 スコープ判断
- [ ] **保留**: プロファイル API は将来 deprecated にするか、恒久的にプリセットとして残すか

> note 1: terse-ui レイヤは terse 本体に含めない。UI 部品が必要な場合は terse-prompt のように **terse を利用する別ライブラリ** として構築する方針。再設計プロポーザル内で「将来構想」として図に含めているのは、terse の責務範囲を明確化する目的のみで、terse 本体の開発スコープには含まれない。

### 4.2 設計詳細

#### 4.2.1 ケイパビリティフラグの粒度【確定】
- **方針**: 現状の粒度（`terse_capabilities_t` の20以上のフィールド）を保ちつつ整理する
- **整理の対象**:
  - 色関連の散在（`has_sgr_basic` / `has_sgr_extended` / `has_truecolor` / `colors` の4箇所）を統合または関係性を明確化
  - 同じ概念が複数フィールドに分散している箇所を見直す
- **段階の追加**: `terse_color_support_t` に 4 色段階を追加し、Human68k を表現可能にする
- **粒度の現状認識**: terse は既に細かいケイパビリティ構造体を持っており、Phase 4 の本質は「ゼロから作る」ではなく「**既存のケイパビリティに要求 API を被せる + 散在の整理 + 段階の追加**」である
- **既存の利用側コードを大きく壊さない**ことを優先

#### 4.2.2 色数の段階分け【確定】
- **方針**: `terse_color_support_t` を以下の 5 段階に拡張する
  ```c
  TERSE_COLOR_NONE
  TERSE_COLOR_BASIC4        /* 新規: Human68k 等 */
  TERSE_COLOR_BASIC16       /* 既存 */
  TERSE_COLOR_PALETTE256    /* 既存 */
  TERSE_COLOR_TRUECOLOR     /* 既存 */
  ```
- **採用根拠**: 具体的な需要（Human68k）に応える最小限の拡張。2 色 / 8 色 / 88 色は現状サポート予定のプラットフォームに該当がない
- **将来の拡張**: 必要が出てきたら段階を追加する（enum 末尾追加なら ABI 互換）

#### 4.2.3 バッファドモードでの色・属性表現（セル構造体のレイアウト）【確定】

**方針**: セル構造体は文字情報 + `terse_color_t` (fg/bg) + 装飾 effects で構成する。

**index 色の表現**:
- `terse_color_t.kind` を `PALETTE` に統一（4 色 / 16 色 / 256 色を `PALETTE` 1 つの kind で表現し、`palette.value` に index を入れる）
- 端末の対応色数は `terse_capabilities_t.colors`（5 段階）で別途持つ
- これにより既存の `BASIC16` / `PALETTE256` の重複が解消される
- 既存 API への破壊的変更を伴うため、移行戦略を別途検討する必要あり

**TrueColor の alpha**:
- **alpha チャンネルは持たない**（RGB 3 バイトのみ）
- 根拠: モダン端末（iTerm2 / Ghostty / kitty / Warp）も**セル単位の alpha 伝送はサポートしていない**。「セルを透過」= 背景色を指定しない (`^[[49m`) で、ウィンドウ全体の透明度がそのまま見える方式
- `(0, 0, 0)` は明確に「漆黒」を表す
- 「未指定（背景を透過）」は `kind = DEFAULT` で表現

**ウィンドウ透明度**:
- セル色とは別 API として将来検討（例: `terse_set_window_opacity()`）
- iTerm2 OSC 1337 / kitty OSC 11 拡張等をプラットフォーム層で吸収
- セル構造体には含めない

**セル構造体（案）**:
```c
struct terse_cell {
    /* 文字情報（~8 bytes）*/
    char     utf8_char[5];
    uint8_t  char_len;
    uint8_t  display_width;
    bool     is_continuation;

    /* 色（terse_color_t をそのまま使用）*/
    terse_color_t fg;   /* DEFAULT=未指定 / PALETTE / TRUECOLOR */
    terse_color_t bg;

    /* 装飾 */
    uint16_t effects;   /* bold/italic/underline 等のビットマスク */
};
```
セルサイズは `terse_color_t` の実サイズに依存（タグ + ユニオン）。

#### 4.2.4 alt screen / scrollback の扱い【確定】

**alt screen サポート**: 提供する。低レベル API とモード化の両方を用意。

- 低レベル API:
  ```c
  terse_error_t terse_enter_alt_screen(terse_handle_t handle);
  terse_error_t terse_leave_alt_screen(terse_handle_t handle);
  ```
- モード化: `terse_options_t` に `use_alt_screen` フラグを追加し、`terse_open()` で自動進入、`terse_close()` で自動退出
- ケイパビリティ: `terse_capabilities_t` に `has_alt_screen` を追加（DEC Private Mode `1049` のサポート可否）
- エスケープシーケンス: 標準は `^[[?1049h` / `^[[?1049l`。互換のため `^[[?1047` / `^[[?47` 系も内部でフォールバックを検討

**scrollback 専用 API**: **提供しない**

- alt screen により大半のスクロールバック汚染問題は解決する
- scrollback 行への能動的なアクセスは端末プロトコル上ほぼ手段がなく、標準化されていない
- ncurses / notcurses 等の主要 TUI ライブラリも同様のスタンス
- 必要が出てきた時点で別途検討

#### 4.2.5 Windows での POSIX 相当機能の提供範囲【確定】

**方針**: Phase 7（POSIX 拡張ヘッダ）の初期版では **POSIX のみ提供**。Windows 拡張は後追いで検討。

- 初期版: `<terse/posix.h>` で `terse_posix_get_input_fd()` 等を提供
- Windows: 既存の `terse_wait_event()` で対応。`<terse/win32.h>` は将来必要が出てから設計
- 根拠:
  - POSIX `pollfd` と Windows `WaitForMultipleObjects` は API 形式が大きく異なり、共通抽象化は難しい
  - Windows ユーザーは多くの場合 `terse_wait_event()` で十分なケースが多いと想定
  - 設計・実装・テストの負荷を分散できる
- 将来検討時の論点: `HANDLE` ベースの統合 API か、`WaitForMultipleObjects` 相当のラッパーか

### 4.3 terse-prompt との関係【確定】

#### 4.3.1 描画コードの取り込み方
- **方針**: **設計借用で再実装**
- 根拠:
  - セル構造体が変わる（`reserved` フィールド → `terse_color_t fg/bg + effects`）ため、そのままコピーできない
  - `terse_handle` 依存への変更、色対応のための SGR 出力追加など、実質書き直しが必要
  - ロジック（diff 計算、バッファ管理）は細部までそのまま参考にできる
  - シンプルな関数は実質コピペに近くなるが、公式には「コピー」とは位置づけない

#### 4.3.2 terse-prompt 側のリファクタリングのタイミング
- **方針**: **ハイブリッド**
- Phase 5（terse バッファドモード導入）内で **最小限のプロトタイプ適用** を terse-prompt 側で行い、API を検証する
- 本格的なリファクタリングは **Phase 5.5** として独立させる
- 根拠:
  - terse のバッファ API を実利用者なしで FIX するのは危険
  - 一方、Phase 5 のスコープを膨らませすぎない
  - Phase 5.5 では terse-prompt の `tprompt_display.c` 汎用部分（500-700 行）を削除して terse バッファ層に置き換える

#### 4.3.3 テストの扱い
- **方針**: terse-prompt のテストを **観点として参考にしつつ、terse 側で新規作成**
- 根拠:
  - セル構造体・API が変わるため、直接移植は不可能
  - テスト**内容**（何を検証しているか）は移植価値が高い
    - 例: backspace 後のカーソル位置、diff 計算の正しさ、リサイズ時の挙動
  - Phase 6 でテスト戦略全体を刷新する予定があるため、移植したテストもいずれ書き直される前提
  - terse-prompt 側のテストはプロンプト固有機能（補完・履歴・マルチライン編集等）として残す

### 4.4 テスト【確定】

#### 4.4.1 ゴールデンファイルの保守フロー
- **方針**: **ハイブリッド** — 手動更新を基本としつつ、フラグで一括更新可能
- 運用:
  - 通常テスト実行時はゴールデンファイルとの比較のみ（書き込まない）
  - `--update-golden` フラグまたは `UPDATE_GOLDEN=1` 環境変数指定時のみ上書き
  - CI では絶対に上書きしない（読み取り専用検証）
  - 上書き後は `git diff` で確認してから commit
- 根拠: 初期構築時の大量更新と、日常運用での「意図せず更新される事故防止」の両立。Rust `insta` / Python `pytest-snapshot` 等で広く採用されているパターン

#### 4.4.2 PTY テストの Windows 対応優先度
- **方針**: **POSIX のみ提供、Windows は後追い**
- Phase 6 初期版: POSIX 環境で `forkpty()` / `openpty()` ベースの PTY テストを提供
- Windows: 通常の ctest（ユニットテスト + ゴールデンテスト）のみ
- ConPTY を使った Windows 版 PTY テストは将来の別 Phase で検討
- 根拠: Section 4.2.5（Windows での POSIX 相当機能の後追い対応）と一貫した判断

#### 4.4.3 端末別スナップショットのカバレッジ
- **方針**: **ケイパビリティの組み合わせベース + 特定端末は専用テスト**
- 基本テスト: 端末名ではなく、ケイパビリティの組み合わせ（色段階 × マウスモード × etc）でカバー
  - 例: 「`colors=BASIC4` のとき正しい SGR を吐く」「`colors=TRUECOLOR` のとき RGB 形式」など
- 特定端末固有機能の専用テスト:
  - iTerm2 の OSC 1337（インライン画像、プロファイル制御）
  - kitty graphics protocol
  - Sixel
  - Human68k 固有挙動
- 根拠:
  - 端末名は時代とともに変わる（プラットフォーム名で固定するとテストが古びる）
  - ケイパビリティの組み合わせを基本にすれば、新しい端末も「既存のケイパビリティ組合せの一例」として扱える
  - 「ゴールデンファイル数の爆発」を抑制できる
